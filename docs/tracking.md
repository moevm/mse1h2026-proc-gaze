# Сервис трекинга

## Назначение

Сервис трекинга обрабатывает запись прокторинга: видео с веб-камеры и видео экрана. Он определяет направление взгляда, накладывает результат на обработанные видео и выделяет участки, которые стоит проверить как подозрительные.

Tracking работает как отдельный воркер. Он получает задание из очереди, читает исходные видео из общего файлового хранилища, выполняет обработку и отправляет результат обратно через RabbitMQ.

## Как проходит обработка

Задание приходит в очередь `tracking.jobs`. В сообщении передаются `recording_id` и относительные пути к двум файлам:

- `path_webcam` — видео с веб-камеры;
- `path_screen` — запись экрана.

Пример входного сообщения:

```json
{
  "recording_id": "12345",
  "path_webcam": "recordings/12345/webcam.mp4",
  "path_screen": "recordings/12345/screen.mp4"
}
```

Воркер открывает оба видео и обрабатывает их покадрово. Основной анализ выполняется по изображению с камеры: сервис ищет лицо, глаза, зрачки, оценивает положение головы и получает направление взгляда. После этого направление взгляда переводится в точку на экране.

Точка дополнительно сглаживается, чтобы на итоговом видео она не прыгала от кадра к кадру. В результате сервис формирует два обработанных файла:

- `camera.mp4` — видео с камеры с разметкой глаз и направлением взгляда;
- `screen.mp4` — запись экрана с наложенной точкой взгляда.

Видео читается и записывается через OpenCV. После обработки временные файлы перекодируются через FFmpeg в формат, который в формат, совместимый с воспроизведением в браузере.

Параллельно сервис собирает подозрительные интервалы. Когда обработка заканчивается, в очередь `tracking.results` отправляется результат: `recording_id`, список интервалов и пути к обработанным видео.

## Основные части

### RabbitMQ-воркер

Точка входа находится в `tracking/main.py`. При старте воркер подключается к RabbitMQ, создает `Tracker` и ожидает сообщения из очереди заданий.

Когда приходит новое задание, обработка запускается через `asyncio.to_thread`, чтобы тяжелая работа с видео не блокировала event loop брокера. После завершения воркер публикует результат в `tracking.results`.

### Tracker

`Tracker` связывает весь пайплайн обработки. Он получает payload задания, открывает видео, запускает анализ кадров, записывает обработанные файлы и собирает итоговый ответ.

Итоговые файлы сохраняются в `results/<recording_id>/`. В ответ попадают относительные пути к этим файлам, чтобы другой сервис мог использовать их без привязки к внутреннему устройству контейнера.


### Анализ взгляда

На кадре с веб-камеры сервис ищет лицо, глаза, зрачки и положение головы, после чего оценивает направление взгляда. Для этой части используются модели OpenVINO и PyTorch: часть пайплайна работает через Intel-модели, а направление взгляда считается через ResNet.

Если в отдельном кадре не удалось получить корректный результат по взгляду, обработка всего видео не останавливается. Такой участок учитывается при формировании подозрительных интервалов.

### Проекция на экран

Направление взгляда само по себе еще не показывает, куда пользователь смотрел на записи экрана. Поэтому оно переводится в точку на плоскости экрана через `GazeMapper`.

После проекции точка проходит через адаптивный фильтр Калмана. Это сглаживает шум модели и мелкие движения, поэтому наложение на итоговом видео выглядит стабильнее.


### Работа с видео

Сервис обрабатывает видео камеры и экрана синхронно, до конца более короткого ролика. На каждом шаге берется кадр камеры и кадр экрана: по камере считается взгляд, а на экран накладывается рассчитанная точка.

Сначала OpenCV пишет временные файлы `camera_raw.mp4` и `screen_raw.mp4`. Затем они перекодируются в `camera.mp4` и `screen.mp4`, после чего временные файлы удаляются.

## Подозрительные интервалы

Сервис не делает финальный вывод о нарушении. Он только выделяет участки записи, которые стоит проверить отдельно.

В результат попадают три ситуации:

- взгляд не найден на кадре камеры;
- найдено несколько взглядов;
- спроецированная точка вышла за пределы экрана.

В коде используются такие причины:

| Значение `description` | Что означает |
| --- | --- |
| `No gaze found on camera frame` | взгляд не удалось определить |
| `Multiple gazes found on camera frame` | найдено несколько взглядов |
| `Projected point went off-screen` | точка взгляда вышла за пределы экрана |

Интервал накапливается, пока подряд идут подозрительные кадры. Когда сервис снова получает корректный результат по взгляду, сервис закрывает текущий интервал и добавляет его в результат с временем начала, длительностью и причиной.

## Формат результата

После обработки сервис отправляет сообщение в `tracking.results`. В нем передаются:

- `recording_id` — идентификатор обработанной записи;
- список подозрительных интервалов;
- путь к обработанному видео камеры;
- путь к обработанному видео экрана.

Пример выходного сообщения:

```json
{
  "recording_id": "12345",
  "intervals": [
    {
      "time": "0:00:12",
      "duration": 3.1,
      "description": "No gaze found on camera frame"
    }
  ],
  "path_processed_webcam": "results/12345/camera.mp4",
  "path_processed_screen": "results/12345/screen.mp4"
}
```
